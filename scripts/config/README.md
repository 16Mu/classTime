# 学校分批上架策略说明

## 📋 概述

本文档说明 `batches.json` 中79所待上架学校的批次划分策略、执行指南和维护方法。

**配置文件位置**: `scripts/config/batches.json`

**数据来源**: 基于 `school-parser.js pending` 命令输出，包含所有有 Extractor 但未在 `schools.json` 中配置的学校。

---

## 🎯 批次划分原则

### 1. 多维度评估体系

批次划分综合考虑以下因素：

| 维度 | 权重 | 说明 |
|------|------|------|
| **学校知名度** | ⭐⭐⭐⭐⭐ | 985/211高校优先，省会城市重点大学次之 |
| **地区集中度** | ⭐⭐⭐⭐ | 同一地区学校归为一批，便于区域性测试 |
| **系统类型一致性** | ⭐⭐⭐⭐ | 相同系统的学校一起处理，便于发现共性bug |
| **Extractor代码质量** | ⭐⭐⭐ | 代码完善、注释清晰的优先 |
| **用户需求量** | ⭐⭐⭐ | 根据反馈和搜索热度判断 |

### 2. 六批次详细方案

#### **Batch 1: 高优先级热门高校（15所）**
- **优先级**: 🔴 高
- **预计工期**: 3天
- **选择标准**:
  - 985/211工程大学（武汉大学、山东大学、哈尔滨工业大学、新疆大学）
  - 省会城市重点本科院校
  - Extractor代码质量较高
- **代表性学校**:
  - `whdx` - 武汉大学（985）
  - `sddx` - 山东大学（985）
  - `hebgydx` - 哈尔滨工业大学（985）
  - `xjdx` - 新疆大学（211）
- **测试重点**:
  - 验证知名高校的课表数据准确性
  - 测试高并发场景下的稳定性
  - 收集用户反馈用于优化

#### **Batch 2: 湖北武汉地区集中（14所）**
- **优先级**: 🔴 高
- **预计工期**: 3天
- **区域重点**: 湖北/武汉
- **特点**: 武汉地区高校密集，便于实地测试和推广
- **学校类型**: 包含本科院校和职业技术学院
- **测试重点**:
  - 区域性网络环境适配
  - 本地化问题排查
  - 批量配置效率验证

#### **Batch 3: 河南+山东华北地区（16所）**
- **优先级**: 🟡 中
- **预计工期**: 4天
- **区域重点**: 河南/山东/华北
- **系统类型**: 以强智（qiangzhi）、树维（shuwei）为主
- **优势**: 系统相对统一，可复用测试用例
- **测试重点**:
  - 强智系统iframe版本兼容性
  - 树维系统多版本适配
  - 华北地区网络延迟处理

#### **Batch 4: 广东+西南地区（15所）**
- **优先级**: 🟡 中
- **预计工期**: 4天
- **区域重点**: 广东/四川/重庆/贵州/广西
- **系统类型多样化**: 乘方、正方、树维、自研系统
- **挑战**: 跨区域、多系统类型
- **测试重点**:
  - 乘方系统API接口验证
  - 西南地区特殊字符编码处理
  - 多系统类型的统一测试框架

#### **Batch 5: 东北+西北+其他地区（10所）**
- **优先级**: 🟢 低
- **预计工期**: 3天
- **区域重点**: 东北/西北/分散省份
- **特点**: 地域分布广，需逐个验证
- **测试重点**:
  - 边远地区网络连通性
  - 特殊时区处理
  - 个别学校的定制化需求

#### **Batch 6: 特殊系统+补充批次（9所）**
- **优先级**: 🟢 低
- **预计工期**: 2天
- **特殊类别**:
  - 移动端专用版本（如 `bjcmzyxymobile`）
  - 通用提取器（`universalsmart`, `schoolschedule`）
  - 变体系统（如 `zjkjxyqingguo` - 青果变体）
- **测试重点**:
  - 移动端适配验证
  - 通用提取器的兼容性
  - 特殊场景的边界测试

## 🚀 执行指南

### 推荐执行顺序

```
Batch 1 (高优) → Batch 2 (湖北) → Batch 3 (河南山东) → Batch 4 (广东西南) → Batch 5 (东北西北) → Batch 6 (特殊)
   ↓              ↓                ↓                   ↓                  ↓                 ↓
  3天           3天              4天                 4天                3天               2天
  ───────────────────────────────────────────────────────────────────────────────────────→
                          总计：19个工作日（约4周）
```

### 并行测试策略

- **同批次内**: 可并行测试 **3-5所学校**
- **跨批次**: 必须按顺序执行（依赖前一批次的经验总结）
- **资源分配**: 
  - 每所学校分配 0.5-1 人日
  - 每批次预留 1 天缓冲时间

### 测试检查清单

#### ✅ 必测项（每所学校）

- [ ] 登录URL可访问性检测
- [ ] 登录页面元素完整性（用户名、密码、验证码）
- [ ] 课表接口可用性和响应速度
- [ ] 数据解析准确性（课程名、教师、教室、时间）
- [ ] 特殊字符处理（中文、括号、空格）
- [ ] 错误场景处理（网络异常、登录失败）

#### ⚠️ 重点项（根据系统类型）

**正方系统 (zfsoft)**:
- [ ] CSRF Token 获取和传递
- [ ] JSON 数据映射正确性
- [ ] 新旧版接口兼容

**树维系统 (shuwei)**:
- [ ] JavaScript 执行环境
- [ ] TaskActivity 解析（版本一 vs 版本二）
- [ ] actTeacherName 字段提取

**强智系统 (qiangzhi)**:
- [ ] iframe 嵌套处理
- [ ] HTML 表格解析准确性
- [ ] 周次表达式转换

**青果系统 (kingosoft)**:
- [ ] 多种课表格式支持（二维表、列表）
- [ ] 登录态保持机制

#### 🔍 可选项（有时间则测试）

- [ ] 并发请求压力测试
- [ ] 不同浏览器兼容性
- [ ] 移动端响应式适配

## 🔧 维护操作指南

### 如何添加新学校

**场景**: 发现新的 Extractor 或需要紧急添加某所学校

**步骤**:

```javascript
// 1. 打开 batches.json
// 2. 找到目标批次（建议插入到当前执行批次的末尾）
// 3. 在 schools 数组中添加学校ID

{
  "batchId": 1,
  "schools": [
    "whdx",
    "sddx",
    // ... 现有学校
    "new_school_id"  // ← 新增学校
  ]
}

// 4. 更新 schoolCount（+1）
"schoolCount": 16  // 原来是15

// 5. 更新统计信息（可选但推荐）
// 6. 记录修改原因（在 notes 或单独的 changelog 中）
```

**示例**: 紧急插入 "pku"（北京大学）到 Batch 1

```json
{
  "batchId": 1,
  "name": "高优先级热门高校",
  "schools": [
    "whdx", "sddx", "hebgydx", "xjdx", "ahjzdx",
    "hnlgdx", "cqust", "ccdx", "shsd", "nmgcjdx",
    "nmgkjdx", "nxsfxy", "sxgckjzydx", "tykjdx",
    "lnzbzzzyjsxy", "pku"
  ],
  "schoolCount": 16,
  "notes": "包含... 紧急新增：北京大学(2026-01-02)"
}
```

### 如何移除学校

**场景**: 某学校无法支持或永久下架

**步骤**:

1. 从对应批次的 `schools` 数组中删除该ID
2. 更新 `schoolCount`（-1）
3. 更新 `totalSchools`（-1）
4. 在 `metadata.excludedSchools` 中记录移除原因

**示例**:

```json
{
  "metadata": {
    "excludedSchools": ["cqvtu", "hnzyydx_yjs", "removed_school"],
    "exclusionReason": {
      "removed_school": "学校已停止使用该教务系统（2026-01-02）"
    }
  }
}
```

### 如何调整批次顺序

**场景**: 优先级变更或资源重新分配

**方案A: 交换两个批次的位置**

```javascript

// 注意：同时更新相关引用
```

**方案B: 将学校从一批次移动到另一批次**


### 如何更新状态

**场景**: 批次开始/完成/阻塞

**状态值**:

| 状态 | 含义 | 使用时机 |
|------|------|----------|
| `pending` | 待处理 | 初始状态或重置后 |
| `in_progress` | 进行中 | 开始执行时 |
| `completed` | 已完成 | 所有学校测试通过 |
| `blocked` | 已阻塞 | 遇到技术难题，等待解决 |
| `skipped` | 已跳过 | 暂不处理（需记录原因） |

**示例**:

```json
{
  "batchId": 1,
  "status": "in_progress",
  "startedAt": "2026-01-02T09:00:00Z",
  "estimatedCompletion": "2026-01-05T18:00:00Z"
}
```

## 🚨 紧急插入机制

### 触发条件

以下情况可启动紧急插入流程：

1. **用户强烈需求**: 多个用户反馈急需某学校
2. **合作院校对接**: 与学校官方有合作关系
3. **Bug修复回归**: 某学校Extractors有重要修复需验证
4. **上级要求**: 项目负责人或产品经理指定

### 插入流程

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│ 提交申请     │ → │ 审批通过     │ → │ 插入配置     │ → │ 通知团队     │
│ (填写理由)   │    │ (负责人)    │    │ (更新JSON)   │    │ (同步信息)   │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
```

### 限制条件

- **每批次最多插入 3 所学校**（避免影响原计划）
- **必须获得审批**（防止随意插队）
- **需记录完整日志**（便于追溯）

### 插入模板

```json
{
  "emergencyInsertion": {
    "schoolId": "xxx",
    "targetBatch": 1,
    "insertionReason": "用户反馈强烈需求",
    "requester": "张三",
    "approver": "李四",
    "insertedAt": "2026-01-02T14:30:00Z",
    "priority": "urgent"
  }
}
```

## ⚠️ 注意事项

### 常见陷阱

1. **不要主观臆断重要性**: 基于客观数据（地区、系统类型、用户反馈）
2. **避免批次过大**: 每批15-20所为宜，便于管理和质量控制
3. **预留缓冲时间**: 每批次额外预留1天应对意外情况
4. **保持文档同步**: 修改配置后及时更新 README 和 progress 文件

### 质量保证

- **代码审查**: 每个 Extractor 配置提交前需经过 Code Review
- **自动化测试**: 尽可能编写单元测试和集成测试
- **用户验收**: 正式上架前邀请目标学校学生内测
- **监控告警**: 上架后持续监控错误率和性能指标

### 回滚策略

如果某学校上架后出现问题：

1. **立即标记**: 在 schools.json 中设置 `"enabled": false`
2. **通知用户**: 通过应用内公告或推送通知
3. **定位问题**: 分析日志，复现 bug
4. **修复验证**: 修复后在内测环境验证
5. **重新上架**: 确认无误后恢复服务
